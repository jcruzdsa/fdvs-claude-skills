Hard boundaries

Guardrails — always enforced

These are not preferences. They prevent real incidents.

🔒
Production schema is off-limits
Never write objects to DB_FANDATA_GOLD_PRD or any schema other than JEFF_DEV_SCHEMA for personal work. ✓ DB_FANANALYTICS.JEFF_DEV_SCHEMA ✗ DB_FANDATA_GOLD_PRD.*
🤝
Shared repo — never overwrite others
In shared repos (dbt_analytics, fandata_dbt), propose merges instead of overwriting someone else's work. Personal proposals stay in the local project folder — never merged without team review.
📋
Stakeholder confirmation before tickets
Confirm names, approval, and justification before creating any Jira ticket or Confluence page. No "draft and delete" — creation is a visible action.
🧪
Run before commit
Always run and validate code locally before any GitHub commit. For Snowflake: redeploy and confirm the app runs at runtime. ✗ ast.parse passing ≠ code works ✓ Redeploy → open → confirm runtime
🔌
Always use MCP for Jira/Confluence
Use mcp__plugin_atlassian_atlassian__* tools. Never curl, fetch, or direct REST calls to Jira/Confluence endpoints.
🏗️
Metadata-first development
Every Snowflake table and column gets a COMMENT. Heavy annotation in dbt YAML. This overrides the default no-comment behavior.
📌
Jira = executable work only
Only create a Jira story if someone will execute it (a DE task, a dbt PR, a validation run). If the purpose is sharing findings with a stakeholder for their review or input, route to Confluence or Slack instead — not a story. Stakeholder reads clutter the backlog and create false velocity. ✓ Colin needs to decide → Confluence/Slack ✗ "Colin review" as a Jira story

Writing conventions
Summary line
  • Verb-first, ≤70 chars
  • ✓ "Add X" "Fix Y" "Migrate Z"
  • No vague phrases ("Placeholder", "TBD", "WIP")
Description must include
  • User story leads every ticket
  • Evidence-based AC (testable, specific)
  • Stakeholder table with roles
Omit: Testing Strategy · Success Metrics · Dependencies (BLOCKED BY/BLOCKS)
Never use: db_bronze or db_silver → use db_fandata_prd
Never reference: Alation (catalog, lineage, or monitoring refs belong in internal docs only)
Style guide links
  • SQL / ingest tickets → SQL Style Guide
  • dbt / AE tickets → dbt Guide
  • Only confirmed URLs — no dead links
  • No Alation refs in ticket body
📄
Every new Epic → paired Confluence page
When you create an Epic, ticket-builder will also create a Confluence page in the same motion. The page must show: business outcome, stakeholder table with approval status, success metrics, timeline, and strategic alignment to FDVS goals. Stories and Tasks don't need a page.
